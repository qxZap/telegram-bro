# IMPORTANT: these are SHELL COMMANDS, not tools.

There is no tool named "telegram", "telegram-send", or anything similar.
You invoke them the same way you'd invoke `ls`, `cat`, or `git`: through
your shell / bash / terminal tool. If you try to call them as if they
were MCP tools or built-in functions, your harness will reject the call.

If your bash tool requires extra fields (e.g. a `description` argument
in some agent harnesses), provide them. If your bash tool fails for any
other reason, tell the user the failure verbatim — DO NOT fake the
behaviour by printing the message to your local stdout. A printed line
in your IDE is NOT a message on the user's phone.

You have a Telegram channel to the user via these terminal commands:

  telegram-send "<text>"                            send a message and exit
  telegram-send -                                   read message body from stdin (use this for anything long, with quotes, URLs, parentheses, or newlines)
  echo "<text>" | telegram-send                     same as above, via pipe
  telegram-send < file.txt                          same, via redirect
  telegram-wait "<text>" [--timeout <sec>]          send a message AND block until the user replies; reply text comes back on stdout
  telegram-wait [--timeout <sec>]                   only block (no send); use only after a previous send in this same flow
  telegram-history [N]                              show the last N messages of the chat
  telegram-schedule "<time>" "<title>" "<text>"     schedule a future message
  telegram-list                                     list scheduled messages
  telegram-cancel <id>                              cancel a scheduled message

Default timeout for telegram-wait is 10 hours. Override with --timeout in seconds.

# When to use stdin

ALWAYS prefer stdin for messages that contain:
  - URLs (the parens/brackets/punctuation are quoting hell on Windows)
  - quotes of either kind, especially in Markdown links
  - newlines
  - more than a sentence or two

The message body is auto-split at 4096 chars with continuation markers, so
length is not your problem — escaping is. Pipe instead of quoting and you
will not have to think about it.

# Identity

Every message you send is auto-prefixed with [<agent-id>] derived from the cwd
(for example E:\eve\eve becomes [e-eve-eve]). telegram-wait only returns
messages addressed to YOUR agent id. The user addresses you by writing
"<agent-id>: <reply>" or by tapping Reply on your message in the Telegram app.
You do not need to insert the prefix yourself.

# Message rules

- Plain text only. No markdown formatting.
- Conversational mode: one short sentence or two per message, like an SMS, no newlines.
- Data dumps (lists, URLs, search results): newlines OK, send via stdin.
- One message per logical event. Do not narrate your reasoning.

# Loop rules — this is where most agents fail. Read carefully.

1. telegram-wait "<text>" sends AND waits in a single call. It is one round trip.
   Never call telegram-send "X" then telegram-wait "Y" — that posts two messages
   in a row before the user has a chance to reply.

2. After you read the user's reply from stdout, your next action is normally
   another telegram-wait "<your next message>" — NOT exiting and NOT printing
   to your local terminal. The conversation continues on Telegram, not in your
   IDE or terminal.

3. Echoing the user's message back ("You said: ...") is forbidden. Reply with
   actual content.

4. Continue the loop until ONE of these is true:
     - The user explicitly says "stop", "done", "bye", "back to terminal",
       "thanks", or any clear close.
     - The user asks you to perform a local task that requires the terminal
       (in which case briefly tell them on Telegram you are switching back,
       then exit the loop).
     - telegram-wait returns exit code 124 (timeout) twice in a row, which
       means the user walked away.

5. Never call telegram-wait without an immediately preceding outgoing message,
   either in this turn (telegram-wait "<text>") or in the same flow as your
   previous turn. Waiting on nothing leaves the user with no idea that you
   expect input.

# Conversational triggers

If the user says any of: "let's chat on telegram", "continue on telegram",
"talk to me on telegram", "move this to telegram", "ping me on telegram",
"chat with me on telegram", or otherwise asks to leave the terminal, enter
the loop above. Keep going until rule 4 fires.

# Setup assumptions

If the commands above are not on PATH, run `bun link` in the skill repo first.
If telegram-send fails with "chat not found", the user must press Start on
their bot in the Telegram app once.

# Common small-model failures (avoid these)

- Calling `telegram` or `telegram-send` as if it were a registered MCP tool.
  These ARE shell commands. Run them through bash.
- Sending two separate messages instead of using telegram-wait "<text>".
  telegram-wait both sends AND waits — using telegram-send first then
  telegram-wait results in two messages back-to-back.
- Echoing the user's reply back ("You said: ..."). Reply with substance.
- Printing fake messages to local stdout when bash fails ("Reply with y...").
  If you cannot run the command, tell the user the bash error directly.
  Inventing a fake interactive reply prompt is worse than admitting failure.
- Quoting long text with embedded URLs/parens through the shell. Pipe via
  stdin instead — see the stdin section above.
